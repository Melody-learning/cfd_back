### 分层职责

| 层级 | 文件 | 职责 |
|------|------|------|
| **路由层** | `app/routers/*.py` | 接收请求、参数校验、调用下层、组装响应 |
| **服务层** | `app/services/*.py` | 认证业务逻辑、JWT 管理 |
| **连接层** | `app/mt5/connector.py` | MT5 会话管理、请求串行化、心跳保活 |

| **数据层** | `app/schemas/*.py` | Pydantic 请求/响应校验 |

    │                            │                                  │
    │                            │ 3. GET /api/trade/balance        │
    │                            │    ?login=954402&balance=10000   │
    │                            │───────────────────────────────────►

### 2.3 交易执行流程（异步确认模式）

> **关键设计**：`dealer/send_request` 的 retcode=0 仅表示"请求已入队"，不代表已成交。
> 必须轮询 `dealer/get_request_result` 获取最终执行结果。

```
  Client                       Gateway                            MT5
    │                            │                                  │
    │  POST /trade/open          │                                  │
    │  {symbol, direction,       │                                  │
    │   lots, sl?, tp?}          │                                  │
    │───────────────────────────►│                                  │
    │                            │ 1. 解析 JWT → mt5_login           │
    │                            │                                  │
    │                            │ 2. GET /api/tick/last             │
    │                            │    获取当前执行价格                 │
    │                            │───────────────────────────────────►
    │                            │◄───────────────────────────────────
    │                            │                                  │
    │                            │ 3. POST /api/dealer/send_request │
    │                            │    {Action:200, Login, Symbol,    │
    │                            │     Type, Volume, PriceOrder...}  │
    │                            │───────────────────────────────────►
    │                            │◄───────────────────────────────────
    │                            │    {answer: {id: 13992}}         │
    │                            │                                  │
    │                            │ 4. 轮询 (每 300ms, 最多 5s)       │
    │                            │    GET /api/dealer/               │
    │                            │    get_request_result?id=13992    │
    │                            │───────────────────────────────────►
    │                            │◄───────────────────────────────────
    │                            │    Retcode=10009 → 成交完成       │
    │                            │                                  │
    │◄───────────────────────────│                                  │
    │  {order, deal, price,       │                                  │
    │   volume, retcode: "10009", │                                  │
    │   message: "成交成功"}       │                                  │
```

**MT5 交易结果码**:
- `10009` — 成交成功 (Done)
- `10008` — 挂单已挂出 (Placed)
- `10006` — 处理中，继续轮询
- 其他 — 交易失败

```
  Client₁      Client₂          Gateway                         MT5
    │             │                │                               │
    │  WS 连接     │                │                               │
    │─────────────────────────────►│                               │
    │             │  WS 连接        │                               │
    │             │────────────────►│                               │
    │             │                │                               │
    │  subscribe  │                │                               │
    │  [EURUSD]   │                │                               │
    │─────────────────────────────►│                               │
    │             │  subscribe     │                               │
    │             │  [EURUSD,      │                               │
    │             │   XAUUSD]      │                               │
    │             │────────────────►│                               │
    │             │                │                               │
    │             │                │  ┌──── 后台轮询任务 ────┐       │
    │             │                │  │ 每 500ms:           │       │
    │             │                │  │ 1. 汇总订阅品种      │       │
    │             │                │  │    {EURUSD, XAUUSD} │       │
    │             │                │  │ 2. GET /api/tick/last│       │
    │             │                │  │    单次批量请求      │       │
    │             │                │  │ 3. 差值比较          │       │
    │             │                │  │    bid/ask 有变化?   │       │
    │             │                │  │ 4. 有变化 → 广播     │       │
    │             │                │  └─────────────────────┘       │
    │             │                │                               │
    │◄─── quote ──────────────────│                               │
    │             │◄── quote ─────│                               │
    │             │                │                               │
```

**关键实现细节**:
- `ConnectionManager` 维护每个 WebSocket 连接的订阅品种集合
- 全局唯一的轮询任务 `_poll_and_broadcast()` 汇总所有客户端的订阅品种
- 通过差值推送（只在 bid/ask 变化时广播）减少无效推送
- 当前 WebSocket 端点**未做鉴权**

## 3. MT5 连接器设计

核心文件：`app/mt5/connector.py`

MT5 Web API 是**有状态服务**，会话绑定在 TCP 连接上。这决定了连接器的所有设计约束。

### 3.1 设计约束

| 约束 | 实现 | 来源 |
|------|------|------|
| 所有请求必须带 `Connection: keep-alive` | httpx 默认 headers | MT5 文档 |
| 使用单一持久 TCP 连接 | `max_connections=1` | MT5 文档 `maxSockets=1` |
| 认证一次后保持连接 | `_connected` 状态标记 | MT5 文档 |
| 每 20 秒 ping 保活 | `_ping_loop()` → `/api/test/access` | MT5 文档 (180s 超时) |
| 请求串行化 | `asyncio.Lock` 保护 `get()`/`post()` | 单连接限制 |

### 3.2 认证流程

```
Gateway                                     MT5
  │  GET /api/auth/start                      │
  │  ?version=3470&agent=AstralWGateway       │
  │  &login=1015&type=manager                 │
  │──────────────────────────────────────────►│
  │◄──────────────────────────────────────────│
  │  {retcode:"0 Done", srv_rand:"d4e0..."}   │
  │                                           │
  │  计算: MD5(MD5(MD5(密码 UTF16) + "WebAPI") │
  │        + srv_rand_bytes)                  │
  │  生成: cli_rand (16 字节随机数)             │
  │                                           │
  │  GET /api/auth/answer                      │
  │  ?srv_rand_answer=8b67...                  │
  │  &cli_rand=34b6...                         │
  │──────────────────────────────────────────►│
  │◄──────────────────────────────────────────│
  │  {retcode:"0 Done",                        │
  │   cli_rand_answer:"...",                   │
  │   version_trade:"1290"}                    │
  │                                           │
  │  验证 cli_rand_answer 确认服务器身份        │
  │  ✅ 认证完成，开始 20s ping 心跳            │
```

### 3.3 重连策略

```
请求失败
  │
  ├── 首次失败 → 自动重连（新建 client + 重新认证）→ 重试请求
  │
  └── 重试后仍失败 → 抛出 MT5ConnectionError → 返回 503
```

Ping 心跳失败时也会触发自动重连。

### 3.4 请求方法

| 方法 | 加锁 | 重试 | 用途 |
|------|------|------|------|
| `get()` | ✅ | ✅ | 普通 GET 请求 |
| `post()` | ✅ | ✅ | 普通 POST 请求 |
| `raw_get()` | ❌ | ❌ | 交易结果轮询（不阻塞其他请求） |

> `raw_get()` 专用于交易结果轮询 (`get_request_result`)，不加锁是为了不阻塞 ping 心跳。

## 4. 本地数据模型

Gateway 本地 SQLite 数据库 (`astralw_gateway.db`) 只保存**最小必要状态**。

### 4.1 表结构

```
┌─────────────────────────────┐     ┌─────────────────────────────┐
│          users              │     │      refresh_tokens          │
├─────────────────────────────┤     ├─────────────────────────────┤
│ id          INTEGER PK      │◄──┐ │ id          INTEGER PK      │
│ email       VARCHAR(255) UQ │   │ │ user_id     INTEGER FK      │──┘
│ mt5_login   INTEGER UQ      │   │ │ token_hash  VARCHAR(64) UQ  │
│ nickname    VARCHAR(128)    │   │ │ expires_at  DATETIME        │
│ created_at  DATETIME        │   │ │ created_at  DATETIME        │
└─────────────────────────────┘   │ └─────────────────────────────┘
                                  │
                                  └─── CASCADE DELETE
```

### 4.2 数据边界

```
  ┌──────────── 数据来源边界 ────────────┐
  │                                      │
  │  本地 SQLite (认证映射层)              │  MT5 服务器 (交易真相源)
  │  ─────────────────────────            │  ─────────────────────────
  │  ✅ email ↔ mt5_login 映射            │  ✅ 账户余额、净值、保证金
  │  ✅ refresh_token 哈希                │  ✅ 持仓、挂单
  │  ✅ 昵称                              │  ✅ 历史成交、历史订单
  │                                      │  ✅ 实时报价、K 线
  │  ❌ 不存储密码                         │  ✅ 交易执行结果
  │  ❌ 不存储交易数据                     │  ✅ 用户密码校验
  │  ❌ 不存储行情数据                     │
  │                                      │
  └──────────────────────────────────────┘
```

> **核心原则**：本地数据库不是交易账本。如需扩展本地存储，需要明确的架构决策。

## 5. 设计决策

### 5.1 价格、金额等精度敏感字段使用字符串

**决策**：所有对外返回的价格、金额、手数、盈亏字段均使用字符串类型。

**理由**：
- 浮点数在 JSON 序列化/反序列化过程中可能产生精度丢失（如 `1.15` 变为 `1.1499999...`）
- 交易系统对价格精度极度敏感，一个小数位的错误可能导致实质损失
- 字符串避免了客户端和服务端之间的精度不一致

**约定**：内部计算时可临时转为浮点数，但对外输出必须是字符串。

### 5.2 K 线聚合在服务端完成

**决策**：MT5 只提供 M1 原始数据，Gateway 在服务端将 M1 聚合为 M5/M15/M30/H1/H4/D1。

**理由**：
- MT5 Web API `/api/chart/get` 只返回 M1 (1分钟) 原始数据
- 如果下发给客户端再聚合，会极大增加网络流量（M1 数据量大）
- 服务端聚合后可以只返回聚合结果，减少传输量

**风险**：大量历史 K 线聚合可能成为性能热点。

### 5.3 交易执行使用异步确认模式

**决策**：交易接口对客户端表现为同步（等待成交结果后返回），但底层使用 `send_request` + 轮询 `get_request_result` 的异步模式。

**理由**：
- MT5 `dealer/send_request` 的 retcode=0 仅表示请求入队，不表示执行完成
- 客户端需要明确的成交结果（成交价、成交量、订单号）
- Gateway 封装了轮询逻辑，对客户端屏蔽了 MT5 的异步复杂性

**参数**：300ms 轮询间隔，5 秒超时。

### 5.4 WebSocket 行情使用轮询广播模式

**决策**：Gateway 通过后台任务每 500ms 轮询 MT5，然后广播给所有 WebSocket 客户端。

**理由**：
- MT5 Web API 没有原生的 WebSocket 推送能力
- 500ms 间隔在延迟可接受范围内（<1s）
- 差值推送（仅 bid/ask 变化时广播）减少无效流量
- 所有客户端的订阅品种合并为单次 MT5 请求，减少 MT5 压力
